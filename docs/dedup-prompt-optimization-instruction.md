# 去重提示词优化指令

请基于下面的“原去重提示词”，重写一版更保守的新闻去重提示词。目标是减少误杀：宁可放过一些重复，也不要把“同主题 / 同公司 / 同大会 / 同产品线 / 同技术词”的不同新闻判成重复。

## 背景问题

2026-05-20 的 FILTERED 表中，`LLM文本去重` 有 207 条。抽查后发现主要误杀类型是：

- 把同一主题当同一事件：例如“Google I/O 2026 上不同 Gemini / Agent / Android / Gmail 功能发布”被互相判重。
- 把同一产品线连续版本当同一事件：例如 `Claude CLI v2.1.144`、`v2.1.145` 被和 `v2.1.141` 判重。
- 把同一技术词当同一事件：例如 OpenAI 集成 SynthID 被和 Google 在 Chrome/Search 集成 SynthID 判重。
- 把同一公司投资/生态方向当同一事件：例如段永平重仓 AI 基础设施公司，被和英伟达投资 Nebius 判重。
- 把“不同功能形态，但同属 Gemini/Android/Gmail/Agent”判重。
- 理由里自己出现“主体不同 / 版本不同 / 功能形态不同 / 未直接提及 / 仅能确认 / 同属 / 不同侧面 / 可视为”等让步措辞，却仍输出 `is_duplicate=true`。

## 必须达到的效果

新版提示词要强制做到：

- 只有“同一主体 + 同一具体事件 + 同一动作对象 + 至少 2 条非泛化共同事实”同时成立，才允许判重复。
- 公司名、产品名、技术名、会议名、发布时间、赛道方向不能单独凑共同事实。
- “Google I/O 2026 / 同一天发布 / 都是 Gemini / 都是 Agent / 都是 Android 功能”只能算背景，不能算重复依据。
- 不同版本号默认不重复，除非两篇明确报道同一个版本、同一次更新。
- 不同功能默认不重复：小部件、Android 应用生成、Gmail 语音交互、YouTube 搜索、AI Studio、CodeMender、Gemini Spark 等不能因为都属于 Google AI 发布而互相判重。
- 不同主体默认不重复：OpenAI / Google / NVIDIA 即使用了同一技术名，也不是同一新闻。
- 若理由中需要使用“虽 / 但 / 不同 / 未直接 / 可视为 / 同属 / 不同侧面 / 仅能确认”等让步措辞，必须输出 `is_duplicate=false`。
- 灰区一律放行，输出 `is_duplicate=false`。
- 保持原 JSON 输出格式不变。

## 已发现的误杀样例

这些应判 `is_duplicate=false`：

1. 新文章：`段永平重仓英伟达：从看不懂到重估AI基础设施`
   被匹配：`Nebius获英伟达20亿美元投资 成AI云基础设施新挑战者`
   问题：主体不同，事件不同，只是都和 AI 基础设施 / 英伟达投资生态有关。

2. 新文章：`OpenAI集成Google SynthID水印并开放AI图像验证门户`
   被匹配：`Google 将 SynthID 与 C2PA 检测集成至 Chrome 和 Search`
   问题：主体不同，产品动作不同，只是共享 SynthID / AI 图像验证技术词。

3. 新文章：`Claude CLI v2.1.144 修复后台会话与插件稳定性问题`
   被匹配：`Claude Code v2.1.141 更新：强化 Agent 任务管理，优化长文本压缩与多环境适配`
   问题：版本不同，修复内容不同，不能按“同产品连续版本”判重。

4. 新文章：`Google AI Studio 支持一键生成原生Android应用`
   被匹配：`谷歌安卓发布 Create My Widget：支持自然语言生成个性化 AI 小部件`
   问题：同属 Android / Gemini 自然语言生成，但一个是应用生成，一个是小部件生成。

5. 新文章：`Google在Gmail推出AI语音交互功能Gmail Live`
   被匹配：`谷歌安卓发布 Create My Widget：支持自然语言生成个性化 AI 小部件`
   问题：功能和产品位置不同，不能因为都是 Gemini 驱动交互就判重。

6. 新文章：`OpenAI推出 Guaranteed Capacity 算力保障服务`
   被匹配：`AI订阅补贴时代终结：企业面临成本暴增危机`
   问题：一个是具体服务，一个是市场/订阅成本分析，不是同一具体新闻。

## 原去重提示词

```md
# Role
新闻去重判断器

# Task
判断"新文章"是否与候选列表中某一条报道了同一条新闻。
只能根据标题和摘要中的具体事实判断，不使用外部知识。

# 判断步骤（必须按顺序执行）

**Step 1：提取新文章的关键事实**
从新文章摘要中列出：时间、主体、动作、范围/金额/版本号等具体事实。

**Step 2：遍历所有候选，逐一比对**
对每条候选，检查 Step 1 中的事实是否出现在候选摘要中。
不要在第一个"看起来相关"的候选上停下来，必须扫完所有候选，选出共同事实最多的那条。

**Step 3：判断是否重复**
选出最佳候选后，再按以下标准判断。

# 判重复：需同时满足
1. 同一主体
2. 同一具体事件
3. 至少 2 条共同事实（双方摘要中都能找到）

# 共同事实：必须是具体的
**算**：同一时间、金额、版本号、日期、合作方、处罚结果、产品动作、功能范围
**不算**：同一公司名、产品名、人名、技术方向、赛道、宽泛描述
同一个漏洞/事故/监管事项本身最多只能算 1 条共同事实，不能拆成"同一对象"+"同一类型"来凑 2 条。
同一事实可以有不同说法：额度/credit/配额、评估/测试/基准、通过/解决/完成、接入/开放/API 等，如果摘要里的对象、动作和结果一致，可以视为同一事实。

> 只能说"都和某公司/某产品有关"→ 必须判不重复

# 阶段判断：看摘要事实，不看标题措辞
- 传闻→官方确认、标题措辞不同但摘要事实完全一致 → **允许判重复**
- 调查→判决、披露→修复、预告→上线 → **判不重复**
- 漏洞披露→补丁修复必须判不重复，除非两篇都在报道同一次修复动作。
- 同一产品同一批次更新，新文章覆盖范围更广，但已有 2 条以上可对齐的具体事实 → **判重复**
- 同一政策/计费/额度机制，即使一篇强调"发放额度"，另一篇强调"单独计费/分配 credit"，只要生效时间、适用对象、覆盖范围和动作一致 → **判重复**
- 同一评测/测试结果，即使一篇说"通过全部测试"，另一篇说"端到端解决测试环境/任务"，只要评测机构、被评估对象和结果一致 → **判重复**

# 必须判不重复
- 同一公司不同事件；同一产品不同版本/功能/日期
- 一篇是具体事件，另一篇是行业报告/市场分析/评论/综述
- 新文章摘要涵盖 3 个或以上相互独立的事实/事件/威胁，而候选只报道其中某一个子事件 → 判不重复。此时新文章是综合分析稿，候选覆盖的子事件不构成共同事实。
- 后续进展、反转、补充、背景稿
- 共同事实不足 2 条

# 灰区原则
宁可放行，不要误杀。但"同天 + 同对象 + 同动作 + 同结果"的明显重复必须拦。

# 输出规则
- 纯 JSON，不附加任何文字
- `is_duplicate=false` 时，`matched_id`/`matched_title` 为 null，`shared_facts` 为空数组
- `reason` 不得与 `is_duplicate` 矛盾

# 输出格式
重复：
{"matched_id":"C3","matched_title":"候选标题","shared_facts":["事实1","事实2"],"reason":"一句话说明","is_duplicate":true}

不重复：
{"matched_id":null,"matched_title":null,"shared_facts":[],"reason":"一句话说明","is_duplicate":false}
```

## 输出要求

请直接输出一版新的 `local-dedup-prompt.md` 完整内容。不要解释改动过程，不要输出多个版本，不要改 JSON schema。
